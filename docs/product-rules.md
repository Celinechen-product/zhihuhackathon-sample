# 人生样本库产品规则

## 产品定位

- 产品名：人生样本库
- 核心价值：用户输入具体人生困惑后，从知乎真实公开内容中找到走过相似困境的人，整理成前人路径和人物样本卡，帮助用户看见真实样本，再自己判断。
- 这不是普通 AI 问答。
- 不替用户做决定。
- 不生成建议结论。
- 不编造人物经历。

## 真实性规则

- 不为了凑满路径或人物而硬生成。
- 路径必须来自真实 people 聚类或分配，不能先编路径再塞人。
- source.url 缺失的内容不能进入正式 people。
- 非作者本人亲历不能进入正式 people。
- 纯观点、鸡汤、攻略、机构号、别人故事默认不进入正式 people。
- 如果结果少，可以返回 1 条路径、1 个人；宁可少，不要假。

## 裸辞路径分配规则

- 裸辞类路径分配按“最新明确结果 > 持续主行动 > 短暂尝试 > 背景/情绪/状态词”判断，不做简单关键词先命中。
- “已接受工作 / 入职 / 新岗位 / 面试后接受工作”等明确回到就业轨道的样本，优先进入“裸辞后重新找工作”。
- “创业 / 自由职业 / 副业 / 自媒体 / 全职交易 / 接项目 / 离开传统职场 / 不再找工作”等作为主行动或最终方向时，进入“裸辞后自由职业试错”。
- “一直休息 / 未积极找工作 / 暂停 / 调整状态 / 考虑转行 / 重新考虑方向 / 找本地企业 / 先停下来”等样本，进入休整再重新开始类路径。
- 如果同时出现“创业”和“找工作”，但最终状态是接受工作或入职新岗位，优先按重新找工作分配；如果最终状态是离开传统职场、继续创业或自由职业，按自由职业试错分配。
- “不上班 / 休息 / 内耗 / 迷茫 / 焦虑 / 想转行 / 想换工作”等弱状态词不能单独触发自由职业路径，也不能压过“找到新工作 / 入职 / 接受工作”等明确结果词。

## Query 前提状态规则

- 对“X 后怎么办 / X 之后怎么办 / X 后怎么生活”类 query，正式 people 要进入 query 已设定的后置状态。
- 判断时同时看正证据和反证据：如“裸辞两年 / 已裸辞 / 裸辞后 / 离职后 / 辞职后 / 待业 / 失业后 / 裸辞后尝试创业 / 辞职后一直休息”等强后置状态证据存在，可以反证 `not_yet_entered_query_premise_state`。
- “准备裸辞 / 想裸辞 / 提出离职 / 尚未正式离职 / 未开始交接 / 离职日期未定 / 交接中”等只有前置阶段证据时，仍过滤为 `not_yet_entered_query_premise_state`。
- 如果正反证据同时出现，优先看是否已有明确“已裸辞 / 已离职 / 裸辞后 / 辞职后”等后置状态；只有明确仍未进入后置状态时才过滤。
- 前提状态证据按字段仲裁：`currentStatus / entryStatus` 中的明确后置结果优先保留，明确前置未完成状态优先过滤；`actionSummary` 同时出现正反证据时，只有“裸辞后顺利找到新工作 / 辞职后一直休息 / 已经裸辞”等明确后置动作能反证前置噪音。
- 不要因为单独出现“离职后 / 裸辞 / 辞职 / 岗位 / 工作”等弱词就 rescue；必须有已经进入后置状态的明确动作、阶段或结果。

## 家庭经济样本规则

- family_money query 中，父母 / 父亲 / 母亲通常是作者发生经济互动的对象，不应仅因文本出现父母就判为 mentioned_person/case。
- 如果作者本人有“给父母钱 / 给生活费或红包 / 转账 / 被父母要钱 / 帮父母还债 / 减少或停止给钱 / 固定金额支持 / 因钱和父母争执”等具体动作，可以作为作者本人亲历，debug 记录 `rescued_by_family_money_author_interaction`。
- 临时要钱、一次性转账、应急支持也可以作为 valid partial_experience；如“父母开口要钱 / 父亲或母亲向我要钱 / 作者转账几千元 / 给了钱”等片段，debug 记录 `rescued_by_family_money_transfer_interaction`。
- 家庭经济类可以是片段经历；只要 situation + actionSummary 有明确金额、转账、红包、给钱、停止给钱等动作，不强制要求 currentStatus 非空。
- 主要讲父母照顾作者、带孩子、做家务，但没有父母要钱、作者给钱或家庭经济支持动作，应过滤为 `family_money_subject_mismatch`。
- 纯建议、理财观点、机构/客户案例、采访转载、朋友同事故事仍然不能进入正式 people。

## 地点迁移一致性规则

- 对 `migration_new_zealand` 这类目标地明确的 query，搜索执行词要尽量带目标地限定，例如“新西兰 WHV / 新西兰打工度假 / 新西兰留学转工签 / 奥克兰生活 / 基督城生活”。
- formal people 必须有明确新西兰目标地证据，如“新西兰 / 奥克兰 / 基督城 / 惠灵顿 / 皇后镇 / 在新西兰 / 去新西兰 / 新西兰 WHV / 新西兰生活 / 从新西兰回来”。
- 只有 WHV、打工度假、留学、工签、移民、旅居等泛机制词，但没有新西兰目标地证据，不能进入正式 people。
- 澳洲、澳大利亚、加拿大、英国、日本、欧洲、美国、新加坡、爱尔兰等非目标地内容，如果没有明确新西兰经历，过滤为 `migration_target_location_mismatch`。
- 新西兰短期旅行、旅游攻略、亲子插班、带孩子短期游学、朋友/客户/孩子故事、纯政策攻略仍然不能进入正式 people。
- “孩子 / 朋友”只作为作者迁移动机、家庭背景或正文顺带提及时，不应单独阻断作者本人迁移经历；亲子插班、带孩子游学、朋友一家/朋友故事、孩子本人经历等主体错配仍然过滤。
- 如果结构化字段明确是作者本人新西兰迁移或生活过程，如移居新西兰、搬到/搬去/搬新西兰、搬到奥克兰、在新西兰奥克兰生活、在新西兰工作、拿到新西兰工作 offer、新西兰工作offer、落地基督城、新西兰 WHV、从新西兰回来等，可以反证 mentioned_person/case 或 subject mismatch，debug 记录 `rescued_by_migration_author_journey`。
- migration rescue 未发生时，debug 要输出 `migration_rescue_blocked_reason`，区分 `reason_not_rescueable`、`hard_negative_matched`、`missing_author_journey_keywords`、`missing_target_location_positive` 等情况。
- search recall debug 需要能看出每个关键词是否执行、raw 召回、去重保留、LLM draft 贡献和 valid people 贡献，用于区分“搜不到”和“过滤光”；如果 `count` 或 max raw 限制导致后续关键词未执行，也要显式标记。

## 表达边界

- “AI 分身”表达必须克制，统一为：基于 TA 公开内容生成的经验问答。
