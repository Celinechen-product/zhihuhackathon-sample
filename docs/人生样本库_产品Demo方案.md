> 当前 Demo 文件：`not-working-new-zealand-demo.html` / `index.html`  
> 当前产品名：**人生样本库**  
> 当前示例主问题：**不工作了之后，我想去新西兰生活**  
> 当前后端地址：`http://127.0.0.1:8000`  
> 前端本地服务建议地址：`http://127.0.0.1:5173/`  
> 本版修改原则：**只补齐产品链路、内容逻辑、边界状态和后端处理规则，不改动现有视觉效果。**

本方案记录当前前端 Demo 的最新产品链路、页面结构、交互方式、后端联调边界和内容生成原则。旧版中的拓扑图、全屏提问弹窗、单一问题样本库、示例 chips、黑色主按钮等设计均已废弃。

## 0. 本版核心原则

### 0.1 产品不是 AI 人生建议

「人生样本库」不是普通 AI 问答，也不是 AI 替用户做人生决定。

产品核心是：

```text
用户输入一个具体人生困惑
→ 系统从知乎真实公开内容中找到走过相似困境的人
→ 把他们的真实经历结构化为「前人路径」和「人物样本卡」
→ 帮助用户通过真实样本理解不同选择、结果和代价
→ 用户自己判断
```

前台不要出现：

- `你应该...`
- `最佳选择是...`
- `系统建议你...`
- `AI 已经帮你判断...`

推荐表达：

```text
你可以先看看真实走过的人，他们做了什么、遇到了什么、后来怎么样。
```

### 0.2 路径必须来自真实内容聚类

「前人路径」不是产品先验写死的答案，而是后端基于真实知乎公开内容聚类后的结果。

正确逻辑：

```text
先检索真实内容
→ 筛选真实经历
→ 抽取经历信息
→ 按选择/行动/结果聚类
→ 生成路径名和一句话解释
```

错误逻辑：

```text
先编 3-5 条路径
→ 再把人塞进去
```

Demo 阶段可以 mock 示例路径，但正式链路必须保证路径从真实内容归纳而来。

### 0.3 样本卡不能为了完整而硬编

知乎内容天然不规整，不是所有回答都有完整时间线。

样本卡要支持三种内容类型：

| 类型 | 内容特征 | 样本卡展示策略 |
|---|---|---|
| 完整经历型 | 有起因、选择、过程、结果 | 可展示较完整时间线 |
| 碎片经历型 | 只提到某个阶段或局部经验 | 展示「公开内容中可确认的经历片段」 |
| 观点中带个人经历型 | 主要是观点，但含个人经历证据 | 弱化时间线，突出「TA 提到的真实经历」和原文入口 |

不要为了视觉统一补写不存在的年份、过程或结果。

### 0.4 “AI 分身”表达要谨慎

Demo 中可以暂时保留现有按钮和页面，但产品方案层面推荐把“AI 分身”解释为：

```text
基于 TA 公开内容生成的经验问答
```

推荐文案：

- `基于 TA 公开内容问问看`
- `先和 TA 的公开经历聊聊`
- `根据 TA 公开分享整理出的回答`

不推荐文案：

- `TA 正在回答你`
- `和 TA 本人聊天`
- `呼叫 TA 的 AI 分身`

除非后续真的获得本人授权或接入真实私信机制，否则不要让用户误解为本人实时回复。

## 1. 产品定位

### 1.1 产品名

```text
人生样本库
```

### 1.2 一句话

用户写下一个真实人生困惑，系统从知乎公开经历中找到走过相似道路的人，让用户看见他们怎么做、读到他们写过的内容，并以低打扰的方式继续追问公开经历。

### 1.3 更准确的产品定义

```text
知乎真实经历的结构化检索与陪伴式阅读产品。
```

也可以对外解释为：

```text
前人经历检索 Agent。
```

但前台不要突出 Agent 炫技。用户真正感知到的应该是：

```text
我看到了真实的人、真实的路、真实的原文。
```

### 1.4 产品边界

- 不直接给标准答案；
- 不替用户做人生决策；
- 不伪装成真人即时回复；
- 不虚构人物经历；
- 不为了凑路径强行归类；
- 不默认真实发送私信或真实发布知乎内容；
- Demo 阶段允许 mock fallback，但真实接口位置已保留。

## 2. 完整用户链路

### 2.1 主链路

```text
首页输入真实人生困惑
→ 系统判断问题是否足够具体
→ 如果足够具体，直接进入 loading
→ 如果过于模糊，可轻量追问 1 个关键问题
→ loading 展示系统正在识别变量、寻找相似经历
→ 后端生成搜索关键词并调用知乎搜索
→ 筛选真实经历内容
→ 抽取经历信息
→ 聚类成前人路径
→ 进入「前人路径」页
→ 用户选择某条路径
→ 同页横向展开该路径下的前人样本库
→ 用户点击某个人物样本
→ 展示人物样本卡
→ 用户阅读 TA 的真实经历摘要和原文入口
→ 用户可进入右侧沉浸式原文阅读
→ 用户可基于 TA 公开内容继续提问
→ 用户可存入「我的样本库」
→ 用户也可发到知乎圈子，召唤更多真实经历
```

### 2.2 不需要追问时

```text
用户输入：不工作之后，我想去新西兰生活
→ 直接进入 loading
→ 识别变量：不工作 / 新西兰 / 长期生活 / 身份与收入 / 职业暂停
→ 检索相似经历
```

### 2.3 需要轻量追问时

适用于用户输入过于宽泛、无法稳定检索真实经历的情况。

示例：

```text
用户输入：我现在很迷茫怎么办
```

这类问题不适合直接生成路径。应该轻量追问：

```text
你更想先看哪类真实经历？

A. 工作/职业方向
B. 亲密关系
C. 家庭关系
D. 去一个地方重新生活
```

追问原则：

- 最多追问 1 次；
- 不做长问卷；
- 不增加明显操作成本；
- 追问不是咨询，而是为了更准确地检索知乎真实经历；
- 用户跳过追问时，也可以用原 query 继续搜索，但结果要更保守。

## 3. 登录与后端联调

### 3.1 前置知乎授权登录

用户访问产品网页时，前端先检查 session 登录状态。

```text
访问前端页面
→ GET http://127.0.0.1:8000/auth/me
→ 若未登录，跳转 http://127.0.0.1:8000/auth/zhihu/login
→ 后端完成知乎 OAuth
→ 后端写入 session cookie
→ 回跳 http://127.0.0.1:5173/
→ 前端再次请求 /auth/me 判断登录态
```

登录成功后：

- 右上角显示用户知乎头像；
- hover 头像显示知乎名称；
- 点击头像，新窗口打开知乎个人主页。

### 3.2 当前搜索接口消费

默认搜索接口：

```text
GET http://127.0.0.1:8000/api/search?query={query}&count=20
```

兼容旧 Demo 接口：

```text
POST http://127.0.0.1:8000/api/demo/search
```

只有设置 `window.LIFE_PATH_USE_DEMO_SEARCH = true` 时才走旧 Demo 接口。

### 3.3 基于公开内容的经验问答接口

当前接口：

```text
POST http://127.0.0.1:8000/api/personas/chat
```

产品语义：

```text
基于 TA 的知乎公开内容生成经验问答。
```

注意：

- 不代表本人实时观点；
- 不代表本人授权回复；
- 不应输出超出公开内容之外的个人信息；
- 不应编造 TA 没有表达过的经历、建议或结论；
- 请求失败时前端可使用本地 mock 回复兜底，但需要在 Demo 讲解中说明这是兜底。

### 3.4 轻量追问判断接口

当前 HTML 尚未实现该模块，后端从 0 搭建时建议预留该接口。

```text
POST http://127.0.0.1:8000/api/query/clarify
```

用途：

```text
首页输入 query 后，先判断是否需要低打扰追问。
```

接口失败时，前端默认不追问，直接进入 Loading 和搜索链路，避免阻塞主体验。

### 3.5 建议后端核心链路

```text
用户 query
→ 问题理解
→ 判断是否需要轻量追问
→ 生成知乎搜索关键词
→ 调用知乎搜索
→ 筛选真实经历
→ 抽取经历信息
→ 聚类成前人路径
→ 生成人物样本卡
→ 返回前端渲染
```

### 3.5 后端返回内容的基本要求

后端返回不要只返回文本摘要，至少需要能支撑这几层关系：

```text
query
paths[]
people[]
sources[]
```

其中：

- `paths`：来自真实内容聚类，不是固定模板；
- `people`：每个人物样本必须关联一个或多个真实知乎来源；
- `sources`：用于还原原文入口、标题、作者、链接、摘录；
- `matchReasons`：解释为什么这个人和用户问题相关；
- `contentType`：标记内容是完整经历、碎片经历，还是观点中带个人经历。

## 4. 首页

### 4.1 首页主结构

首页是一个居中的输入产品，不再是营销页。

首屏内容：

```text
有什么让你纠结了很久的事？越具体越好

[大输入框]
```

首页只保留：

- 顶部导航；
- 产品名和小人 Logo；
- 一行主标题；
- 一个大输入框；
- 输入框右下角发送箭头。

已删除：

- `知乎出品 · 找到走过相似道路的人`
- `写下一个真实困惑...`
- `寻找走过的人` 文字 CTA
- 示例问题 chips
- 首页底部文案

### 4.2 输入框底纹词

输入框内轮换展示底纹词：

```text
不工作了之后，我想去新西兰生活
为了工作，异地恋值得吗？
有什么值得白领尝试的体力劳动
30岁裸辞后怎么办
```

交互：

- 如果用户未输入，点击发送箭头时，当前轮换到底纹词作为 query；
- 如果用户点击输入框，底纹词消失，用户可自行输入；
- 用户点击发送并真正开始搜索时，记录：

```js
state.hasSubmittedQuestion = true;
state.submittedQuestion = state.question;
```

该记录用于后续「我的样本库」按问题归档。

### 4.3 首页视觉

- 使用 Intercom 暖奶油色视觉；
- 首页主内容用 flex 垂直居中；
- 标题和输入框整体位于视口视觉中心附近；
- 输入框高度较高，底纹词轻、细、不抢主操作。

> 本版不改动首页视觉，只补充产品链路规则。

## 5. 首页输入后的轻量追问机制

> 本节是本次补充的核心。追问是搜索前的低打扰补充，不是咨询问卷，也不是新增一页。

### 5.1 追问定位

追问只服务于一件事：

> 当用户输入过于模糊、检索变量不足，导致系统难以稳定找到知乎真实经历时，用 1 个低打扰问题补足检索变量。

追问不是：

- AI 人生诊断；
- 心理咨询问卷；
- 个人画像采集；
- 多轮 Agent 对话；
- 让用户先把人生问题讲完整。

追问的前台感知应该是：

```text
为了帮你找得更准，我想再确认一句。
```

### 5.2 触发时机

用户在首页输入 query 并点击发送箭头后，不应总是立刻进入 Loading。

新的链路：

```text
首页输入 query
→ 点击发送箭头
→ 前端请求 /api/query/clarify 做问题理解和追问判断
→ 如果 needClarification = false：直接进入 Loading 并搜索
→ 如果 needClarification = true：首页输入框下方原地浮出轻量追问卡
→ 用户选择选项 / 点击跳过
→ 进入 Loading
→ 调用搜索接口，带上 query + clarificationAnswer
```

### 5.3 是否追问的判断

不需要追问的情况：

- 用户问题已经包含明确对象、场景或选择；
- 能稳定生成知乎搜索关键词；
- 能检索到相似真实经历；
- 继续追问只会增加操作负担。

例子：

```text
30岁裸辞后怎么办？
为了工作，异地恋值得么？
出来工作后，父母要求给他们钱怎么办？
不工作之后，我想去新西兰生活。
```

需要追问的情况：

- 问题太抽象；
- 缺少人生场景；
- 检索关键词不稳定；
- 可能出现大量泛鸡汤内容；
- 不追问会导致路径聚类失真。

例子：

```text
我很焦虑怎么办？
我现在很迷茫。
我不知道自己要什么。
我想换一种生活。
```

### 5.4 追问卡出现位置

追问卡不跳页、不弹窗、不遮罩。

位置：

```text
首页输入框下方，原地浮出
```

视觉要求：

- 不改动首页整体视觉；
- 沿用当前 Intercom 暖奶油色风格；
- 使用白色 / 半透明白色卡片；
- 浅边框、轻阴影；
- 选项按钮使用弱边框；
- 主流程不被打断；
- 用户可一键跳过。

### 5.5 追问卡内容结构

结构：

```text
为了找到更接近的真实经历，想再确认一句：

{clarificationQuestion}

[选项 A] [选项 B] [选项 C]

[跳过，先帮我找找看]
```

示例 1：

```text
为了找到更接近的真实经历，想再确认一句：

你现在更接近哪种状态？

[已经想做一件事，但不确定代价]
[有几个选择，不知道怎么比较]
[只是想看看有没有人和我一样]

跳过，先帮我找找看
```

示例 2：

```text
为了找到更接近的真实经历，想再确认一句：

你更想看哪类真实经历？

[工作选择]
[关系困扰]
[家庭压力]
[换城市/换国家生活]

跳过，先帮我找找看
```

### 5.6 追问交互规则

用户点击某个选项后：

```text
保存 clarificationAnswer
→ 进入 Loading
→ 调用搜索接口
→ 搜索请求带上 query + clarificationAnswer
```

用户点击跳过后：

```text
clarificationAnswer = ""
→ 进入 Loading
→ 使用原始 query 搜索
```

其他规则：

- 追问最多出现 1 次；
- 不连续追问；
- 不要求用户输入长文本；
- 不强制用户回答；
- 接口失败时默认不追问，直接进入 Loading；
- 追问卡出现后，用户仍可修改首页输入框内容；
- 如果用户修改了 query，应该清空已有 clarificationAnswer，并重新做追问判断。

### 5.7 追问接口协议

新增接口：

```text
POST http://127.0.0.1:8000/api/query/clarify
```

请求：

```json
{
  "query": "我现在很迷茫"
}
```

返回：

```json
{
  "needClarification": true,
  "clarificationQuestion": "你更想看哪类真实经历？",
  "clarificationOptions": [
    "工作选择",
    "关系困扰",
    "家庭压力",
    "换城市/换国家生活"
  ]
}
```

不需要追问时：

```json
{
  "needClarification": false,
  "clarificationQuestion": "",
  "clarificationOptions": []
}
```

### 5.8 追问后的搜索接口

搜索接口需要兼容 clarification 参数。

推荐请求：

```text
GET /api/search?query={query}&clarification={clarificationAnswer}&count=20
```

或者后端也可以支持 POST：

```json
{
  "query": "我现在很迷茫",
  "clarification": "工作选择",
  "count": 20
}
```

后端使用规则：

- query 是用户原始问题；
- clarification 是用户补充变量；
- 生成知乎搜索关键词时，必须同时考虑 query 和 clarification；
- 但前端展示标题仍优先展示用户原始 query，不要把补充选项硬拼成标题。

### 5.9 追问文案原则

推荐：

- 为了找到更接近的真实经历，想再确认一句；
- 你现在更接近哪种情况？
- 你更想看哪类真实经历？
- 跳过，先帮我找找看。

不推荐：

- 请完善你的人生情况；
- AI 需要更多信息才能判断；
- 你应该先明确自己的真实需求；
- 请填写个人背景；
- 系统将基于你的画像进行分析。

### 5.10 前端新增状态

当前 HTML 尚未实现首页轻量追问模块，需要新增以下状态：

```js
state.needClarification
state.clarificationQuestion
state.clarificationOptions
state.clarificationAnswer
state.clarificationShown
state.isCheckingClarification
```

### 5.11 前端新增函数

```js
requestClarification(query)
renderClarificationCard()
submitClarification(answer)
skipClarification()
startSearchWithClarification()
```

注意：

- 不要改动已有视觉系统；
- 不要改动路径页、样本卡、样本库视觉；
- 只在首页输入框下方增加轻量追问卡；
- 如果接口失败，直接走原始搜索链路。

## 6. Loading 页

主文案：

```text
正在寻找可能与你有相似经历的知友……
```

保留步骤：

```text
读取问题
识别核心变量
生成待确认问题
匹配公开经历
```

建议补充产品含义：

| 步骤 | 实际含义 |
|---|---|
| 读取问题 | 解析用户输入的人生困惑 |
| 识别核心变量 | 抽取年龄、关系、地域、职业、身份、现金流等变量 |
| 生成待确认问题 | 判断是否需要轻量追问；若不需要则跳过追问 |
| 匹配公开经历 | 用关键词检索知乎公开内容，并筛选真实经历 |

视觉：

- 继续使用暖浅 Intercom 风格；
- 不使用暗黑科技风；
- 人来人往动效继续使用之前的小人资源。

> 本版不改动 loading 视觉。

## 7. 前人路径页

### 7.1 页面标题

```text
关于「{用户输入的问题}」，TA 们是这样做的...
```

### 7.2 路径卡橱窗

页面初始展示 3-5 张路径卡，不恢复拓扑图、放射路线或关系图。

Demo 示例路径：

1. 先去一年试生活｜128 人走过
2. 留学转工签｜76 人走过
3. 去了又回来｜61 人走过
4. 技术迁移留下来｜54 人走过
5. 远程工作旅居｜43 人走过

正式链路要求：

- 路径数量不固定，建议 3-5 条；
- 如果只有 1 条有效路径，可直接进入该路径下的人物样本库；
- 如果有 2 条路径，可以展示 2 张路径卡，不强行补到 3 条；
- 如果有 5 条以上路径，优先展示人数最多或证据最强的 5 条；
- 路径按「走过的人数」或「证据强度」排序；
- 路径名要中性，不要替用户判断好坏。

每张路径卡只展示：

- 序号 + track name；
- `xx 人走过`；
- 一句话解释；
- 底部最多 4 个前人头像。

不展示：

- `适合你看...`
- hover 常见问题；
- 页面级底部出口；
- 拓扑图。

### 7.3 路径命名规则

路径名应该描述「前人实际做了什么」，不要描述「用户应该怎么做」。

推荐：

```text
先去一年试生活
留学转工签
去了又回来
技术迁移留下来
远程工作旅居
```

不推荐：

```text
最稳妥路线
高成功率路线
适合你的路线
别冲动路线
```

一句话解释结构：

```text
他们大多先做了什么，过程中遇到什么，后来走向哪里。
```

### 7.4 路径卡内部排版

路径卡内部结构：

```html
<button class="path-card">
  <div class="path-card-header">
    <span class="path-index">01</span>
    <h2>先去一年试生活</h2>
  </div>
  <span class="path-count">128 人走过</span>
  <p class="path-desc">路径解释</p>
  <span class="path-card-peeps">头像</span>
</button>
```

排版规则：

- 序号 badge 和路径名第一行顶部对齐；
- 标题区固定高度，保证 5 张卡的 `xx 人走过` 顶部统一；
- 路径名到人数间距约 18px；
- 人数到解释间距约 12px；
- 底部头像固定在底部，不参与文字布局。

### 7.5 路径点击后的展开

点击某张路径卡后，不跳转新页面。

当前交互：

```text
初始状态：只展示 5 张路径卡
点击某路径卡
→ 在 5 张路径卡下方出现该路径的人物样本库
→ 页面自动滚动到下方样本库
→ 用户上滑仍能看到上方 5 张路径卡
→ 点击另一张路径卡，下方样本库内容替换为新路径
```

不再使用：

- 独立 `samples` 页面作为路径主入口；
- `收起，看看其他路径` 按钮；
- 点击后整页闪跳。

展开区标题：

```text
{路径名}的人
```

副标题：

```text
该路径的一句话解释
```

## 8. 路径内前人样本库

### 8.1 人物舞台

路径展开后，展示横向人物样本舞台。

要求：

- 人物头像优先来自后端检索返回的真实用户头像；
- 若后端字段缺失，前端使用 mock fallback；
- 中央人物可出现气泡；
- 点击头像后打开该人物样本卡；
- 其他人物弱化或收起，保持舞台空间。

### 8.2 人物样本卡

人物样本卡信息顺序：

1. TA 是谁
2. 为什么匹配到 TA
3. TA 的经历时间线
4. TA 的真实经历
5. 底部操作按钮

人物名称：

- 使用后端返回的知乎昵称；
- 可点击；
- 新窗口打开对应知乎主页。

已删除：

- `前人样本｜路径名`
- `TA 留下的关键经验`
- `再看看其他人`
- 样本卡内的知乎圈子出口。

底部按钮：

```text
向 TA 提问
存入我的样本库
```

### 8.3 样本卡降级展示

不是所有人物样本都有完整时间线。样本卡必须根据内容完整度降级。

#### A. 完整经历型

展示：

```text
TA 是谁
为什么匹配到 TA
TA 的经历时间线
TA 的真实经历
```

#### B. 碎片经历型

如果只有局部经历，时间线标题可保持不变，但内容只展示可确认片段：

```text
公开内容中可确认的经历片段
- 2022 年前后：TA 提到自己曾经裸辞
- 后续：TA 提到尝试过短期旅居
```

不要补齐不存在的起点和终点。

#### C. 观点中带个人经历型

如果主要是观点，但包含个人经历，样本卡重点展示：

```text
TA 提到的个人经历
TA 对这类问题的看法
知乎原文入口
```

时间线可以弱化为：

```text
公开内容中未形成完整时间线
```

### 8.4 为什么匹配到 TA

匹配理由不要写成泛泛的“你们都很迷茫”。

推荐结构：

```text
- TA 的公开内容中出现了与你问题相同的核心场景：不工作 / 新西兰 / 长期生活
- TA 真实经历过类似选择：先暂停工作，再尝试海外生活
- TA 的原文中提到过关键代价：现金流、身份周期、工作边界
```

### 8.5 TA 的真实经历

原 `TA 写过的原文入口` 改为：

```text
TA的真实经历
```

原文入口不做完整卡片，只保留文本与暖黄色虚线下划线，表示可点击。

每条真实经历至少应保留：

- 原文标题；
- 作者昵称；
- 原文链接；
- 内容类型；
- 摘录或摘要；
- 是否为完整经历型内容。

## 9. 阅读原文与提问面板

### 9.1 阅读原文

点击某篇真实经历后，使用右侧沉浸式展开。

布局：

```text
左侧：人物样本卡收窄
右侧：原文阅读器大面板
```

不跳转新页面，不出现全屏弹窗。

### 9.2 原文、AI 摘要、人物卡的关系

三者关系必须讲清楚：

```text
知乎原文 = 真实来源
人物样本卡 = 基于公开内容的结构化摘要
经验问答 = 基于公开内容生成的辅助理解
```

优先级：

```text
原文 > 摘要 > 经验问答
```

当摘要和原文可能有偏差时，以原文为准。

### 9.3 向 TA 提问

点击人物样本卡里的：

```text
向 TA 提问
```

不再打开独立页面，也不再打开全屏弹窗。

当前交互：

```text
人物样本卡左移并收窄
右侧展开提问面板
右侧面板和阅读原文共用沉浸式布局
```

右侧提问面板内容：

- 顶部弱提示：`向 TA 提问`
- 一个大输入框；
- 底部按钮：

```text
基于 TA 公开内容先问问
递给 TA 本人
先不问，我再看看
```

说明：

- 如果当前 Demo 暂时不改前端按钮，可保留现有 `呼叫 TA 的 AI 分身先聊聊`；
- 正式产品文案建议替换为 `基于 TA 公开内容先问问`；
- `递给 TA 本人`：Demo 阶段只 toast，不真实发送；
- `先不问，我再看看`：弱文字按钮，收起右侧提问面板。

右侧区域互斥：

```text
readerOpen 和 askPanelOpen 不会同时打开
```

## 10. 基于公开内容的经验问答页

入口：

- 从右侧提问面板点击 `基于 TA 公开内容先问问`；
- 从我的样本库点击 `继续问公开经历` 或当前 Demo 的 `继续问 AI 分身`。

聊天页左侧展示：

- 对应前人的头像卡；
- 前人昵称；
- 路径信息；
- 说明：回答基于 TA 的公开内容生成，不代表本人实时观点。

聊天消息：

- 用户问题来自提问面板输入框；
- 后端接口可用时走 `/api/personas/chat`；
- 接口失败时使用本地 mock 回复兜底。

### 10.1 经验问答的回答边界

回答必须遵守：

- 只能基于 TA 的公开内容和原文摘要回答；
- 不得编造 TA 的身份、收入、城市、家庭信息；
- 不得冒充本人语气说“我现在建议你”；
- 可以表达为“从 TA 公开分享看，TA 当时遇到的是……”；
- 如果公开内容不足，应明确说“TA 的公开内容里没有提到这一点”。

推荐回答格式：

```text
从 TA 的公开分享看，TA 当时主要遇到三个问题：...

但 TA 没有明确提到你问的这个细节，所以这里不能替 TA 补充。

你可以回到原文重点看这几段：...
```

## 11. 我的样本库

### 11.1 初始状态

初始状态：

```js
state.savedQuestionGroups = [];
state.hasSubmittedQuestion = false;
state.submittedQuestion = "";
```

用户刚进入首页、未完成搜索、未保存样本时，点击 `我的样本库` 显示空状态。

不会因为以下原因生成样本库内容：

- 首页有默认底纹词；
- mock people 数据存在；
- 用户进入首页；
- 用户进入路径页；
- 用户没有点击 `存入我的样本库`。

### 11.2 空状态

空状态文案：

```text
你还没有保存样本
当你遇到一个想继续参考的人，可以把 TA 收进这里。
```

按钮：

```text
去寻找走过的人
```

点击后回到首页。

### 11.3 按问题分组

保存样本后，样本库按用户已提交的问题分组。

结构：

```text
我的样本库

“问题 A”  已保存 2 个样本
  样本 1
  样本 2

“问题 B”  已保存 3 个样本
  样本 1
  样本 2
  样本 3
```

数据结构：

```js
state.savedQuestionGroups = [
  {
    id: "q_xxx",
    question: "不工作了之后，我想去新西兰生活",
    createdAt: 1715673600000,
    lastSavedAt: 1715677200000,
    samples: [
      { personId: "yu", savedAt: 1715677200000 }
    ]
  }
];
```

排序：

- 问题组按 `lastSavedAt DESC`，再按 `createdAt DESC`；
- 组内样本按 `savedAt DESC`。

### 11.4 样本库长期价值

「我的样本库」不是单次收藏夹，而是长期人生问题归档。

后续正式版可以演进为：

```text
问题 → 保存过的人 → 看过的原文 → 问过的问题 → 后续回看
```

但 MVP 阶段只需要做到：

- 按问题归档；
- 支持回看 TA 的经历；
- 支持继续基于公开内容提问；
- 不因为 mock 数据误生成归档。

### 11.5 样本库视觉

当前视觉原则：

```text
问题组是卡片，样本是列表
```

问题组：

- 外层白色卡片；
- 标题显示问题文本；
- `已保存 x 个样本` 做成弱标签；
- 长问题单行省略。

样本：

- 不再是卡片里套卡片；
- 改成无边框列表行；
- 样本之间用浅分割线；
- hover 不出现特殊底色；
- 左侧头像；
- 中间昵称、路径、概要；
- 右侧两个边框按钮。

样本行按钮：

```text
回看 TA 的经历
继续问 AI 分身
```

> 正式文案建议将 `继续问 AI 分身` 调整为 `继续问公开经历` 或 `继续基于公开内容问问`。如果当前前端暂不改文案，可先保留。

## 12. 知乎圈子出口与草稿页

路径展开状态下，在样本库面板下方展示页面级出口：

```text
想问问更多人？
生成一条知乎想法，等待更多路过的人。

发到「人生样本库」圈子
```

要求：

- 不放进人物样本卡；
- 不是核心动作；
- 按钮使用弱边框，不抢 `向 TA 提问`。

点击后进入草稿页。

草稿页只保留：

- 标题输入框，label：`标题`
- 正文输入框，label：`正文`
- 按钮：`发布`
- 按钮：`返回样本库`

已删除：

- 推荐发布圈子；
- 说明文案；
- `再改改` 按钮；
- `去知乎圈子发布` 文案。

### 12.1 圈子出口的产品定位

圈子出口不是主链路，而是低召回或想要更多真实经历时的补充路径。

适合出现的时机：

- 用户已经看完一条路径下的人物样本；
- 用户没有找到完全匹配的人；
- 用户想听更多真实经历。

不适合：

- 首页直接强推发圈子；
- 样本卡内强插发圈子；
- 把发圈子包装成必做动作。

## 13. 低召回与边界状态

### 13.1 没有搜到足够真实经历

不要硬编路径。

推荐展示：

```text
这次没有找到足够相似的公开经历。
你可以换一种更具体的说法，或者发到知乎圈子问问更多人。
```

可提供：

- 修改问题；
- 发到知乎圈子；
- 查看较宽泛的相关经历。

### 13.2 只有 1 个有效人物

不需要展示路径选择中转。

推荐链路：

```text
loading
→ 直接进入该人物样本卡 / 该路径下样本库
```

页面标题可改为：

```text
我们先找到了一位经历相近的人
```

### 13.3 只有 1 条路径，但有多个人

直接进入该路径的人物样本库，不展示多路径橱窗。

标题：

```text
这些人走过一条相近的路
```

### 13.4 有 3 条路径，每条只有 1 人

可以展示路径卡。

原因：

- 路径差异仍然有意义；
- 用户可以先理解不同选择方向；
- 点击后每条路径只展示 1 个样本，不能显得像 bug。

样本库标题建议：

```text
这条路上，我们先找到 1 位公开分享过的人
```

### 13.5 文案过长

需要限制：

| 字段 | 建议限制 |
|---|---|
| 路径名 | 8-12 个中文字符，最多 16 字 |
| 路径一句话解释 | 35-60 字 |
| 人物 oneLine | 35-70 字 |
| 匹配理由 | 2-3 条，每条不超过 28 字 |
| 时间线事件 | 每条不超过 32 字 |
| 原文标题 | 超长省略，不换成两行以上 |

### 13.6 内容可信度不足

如果内容不是明确个人经历，只是观点，应标记为：

```text
观点中包含个人经历
```

不要展示成完整人生故事。

整体保持 Intercom 暖米色 / 浅奶油色方向。

原则：

--bg: #FBF7EF;

## 16. Demo 演示话术

推荐讲法：

```text
这个产品不是让 AI 直接告诉你人生该怎么选。

我们做的是：当用户输入一个具体人生困惑时，系统会去知乎公开内容里找真实走过相似处境的人，把他们的经历整理成几条「前人路径」。

用户不是看一个 AI 答案，而是先看到不同人真实做过什么、遇到什么、后来怎么样。

每个人物样本都能回到知乎原文，AI 只负责结构化整理和辅助阅读，不替用户下结论。
```

一句话总结：

```text
AI 不替你决定人生，它帮你找到走过的人。
```

## 17. 对后端同学的合并提醒

这版主要是产品逻辑、内容规则和边界状态更新，原则上不要求改动视觉效果。

但后端需要注意：

1. 路径必须来自真实内容聚类，不要先验写死；
2. 样本卡字段要允许缺失，不要强制完整时间线；
3. 每个人物样本必须关联真实知乎原文来源；
4. 经验问答只能基于公开内容回答；
5. 低召回时宁可提示用户换问法或发圈子，也不要硬编路径；
6. 如果当前前端 HTML 已经被改过，不要直接全量覆盖，先合并逻辑差异。

如果后端同学已经基于旧版 HTML 做过本地联调修改，请按模块合并：

- 首页居中；
- 右侧沉浸式提问面板；
- 样本库按问题分组；
- 路径卡内部排版；
- 路径生成与样本卡降级规则；
- 公开内容经验问答边界；
- 低召回和边界状态。
