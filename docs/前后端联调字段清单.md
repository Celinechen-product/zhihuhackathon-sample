| 字段 | 类型 | 必需 | 当前后端 | 用途 |
|---|---|---:|---|---|
| `id` | string | 是 | 缺失 | 人物样本唯一 id，不建议直接用内容 id |
| `name` | string | 是 | 可从 `author.name` fallback | 人物名/作者名 |
| `pathId` | string | 是 | 缺失 | 关联路径 |
| `role` | string | 是 | 缺失 | 人物身份摘要，如“前互联网运营” |
| `badge` | string | 是 | 缺失 | 移动人物标签里的关键经历 |
| `gender` | string | 可选 | 缺失 | 可不传，前端不再展示显眼性别徽章 |
| `avatar` | string | 可选 | 可从 `author.avatar` fallback | 人物形象/头像 |
| `oneLine` | string | 是 | 缺失 | 样本卡核心句 |
| `who` | string | 是 | 缺失 | `TA 是谁` 段落 |

| `timeline[]` | object[] | 是 | 缺失 | 经历时间线 |
| `lesson` | string | 是 | 缺失 | 关键经验/takeaway |

## 7. 缺失清单：`articles[]`

当前后端 `items[]` 可以作为文章/回答来源，但还没有挂到人物样本下。推荐每个 `people[].articles[]` 返回：

| 字段 | 类型 | 必需 | 当前后端 | 用途 |
|---|---|---:|---|---|
| `id` | string | 是 | 可用 `items[].id` | 原文唯一 id |
| `title` | string | 是 | 可用 `items[].title` | 人物卡原文入口标题 |
| `author` | string | 建议 | 可用 `author.name` | 阅读器 meta |
| `sourceName` | string | 建议 | 可由 `type` 映射 | 如“知乎回答” |
| `sourceUrl` | string | 是 | 可用 `url`/`source.url` | 查看知乎原文 |
| `publishedAt` | string | 建议 | 可由 `editTime` 映射 | 时间展示 |
| `summary` | string | 建议 | 可用 `text`/`evidence.text` | 摘要 |
| `body[]` | string[] | 可选 | 当前没有全文 | 页内阅读器正文段落 |

如果合规上不能返回全文，`body` 可以为空；但至少要有 `title + sourceUrl + summary`，阅读器可展示摘要并保留跳转知乎原文。

## 9. 给后端 Codex 的建议实现路径

### 最快联调方案

保留现有 `GET /api/search` 作为原始知乎搜索接口，同时新增一个产品层接口：

```http
POST /api/demo/search
Content-Type: application/json
```

请求：

```json
{
  "query": "不工作了之后，我想去新西兰生活",
  "count": 20,
  "mode": "mock_or_real"
}
```

响应直接返回：

```json
{
  "queryId": "query_123",
  "query": "不工作了之后，我想去新西兰生活",
  "analysis": {},
  "paths": [],
  "people": []
}
```

这样不会破坏当前 P0 搜索接口，也更适合长 query 和大模型编排参数。

### 今日最低可跑通方案

如果后端今天来不及接大模型，可以先用固定 mock/规则把 `items[]` 转成 `paths[] + people[] + articles[]`：

1. `GET /api/search` 负责召回知乎内容。
2. 后端根据 query 生成 3 到 5 个 mock path。
3. 把每个 `item` 包成一个 `person`，先用 `author.name/avatar` 和 `title/text/url` 兜底。
4. `timeline/overlaps/lesson/who/badge` 可以先由规则或 mock 生成，字段先稳定。

### 中期真实方案

1. 用户 query 进入意图识别模型，产出 `analysis` 和搜索关键词。
2. 调知乎 API 召回多个内容。
3. 模型对内容做路径聚类，产出 `paths[]`。
4. 模型从内容中抽取/总结前人样本，产出 `people[]`。
5. 每个样本绑定真实 `source/evidence/articles`，保证可追溯。

## 11. Intercom 风格新版链路新增字段影响

产品方案已新增三类链路：轻量登录提醒、知乎圈子提问草稿、我的样本库小人卡。这些不一定都要今天后端实现，但需要提前对齐字段边界。

### 11.1 登录状态

MVP 可以前端模拟登录；如果后端接入，需要一个轻量 session/profile 接口。

建议返回：

```json
{
  "loggedIn": true,
  "user": {
    "id": "zhihu_user_123",
    "name": "知乎用户",
    "avatar": "https://..."
  },
  "matchSignalsAvailable": true
}
```

字段用途：

| 字段 | 用途 |
|---|---|
| `loggedIn` | 控制右上角显示「登录」还是知乎头像 |
| `user.name` | 头像 hover / 账号菜单 |
| `user.avatar` | 右上角头像 |
| `matchSignalsAvailable` | 解释登录后是否能增强匹配 |

### 11.2 知乎圈子提问草稿

新版后链路新增页面级出口：`发到知乎圈子问问更多人`。它不是某个人物卡的动作，而是围绕用户原始问题生成公开提问草稿。

建议接口：

```http
POST /api/community/draft
```

请求：

```json
{
  "queryId": "query_123",
  "query": "不工作了之后，我想去新西兰生活",
  "pathId": "try-one-year",
  "viewedPersonIds": ["person_001", "person_002"]
}
```

返回：

```json
{
  "draftId": "draft_123",
  "recommendedCircles": ["新西兰生活", "海外生活", "Gap一年", "不工作之后", "职业转向"],
  "title": "不工作之后想去新西兰生活，怎么判断自己是真的适合，还是只是想逃离现在？",
  "body": "最近我一直在想一个问题...",
  "sourcePersonIds": ["person_001", "person_002"]
}
```

如果后端今天不做，前端可以先本地生成草稿。

### 11.3 发布到知乎圈子

正式发布必须由用户确认，不能自动发布。

建议后端至少明确：

| 能力 | 今日是否需要 |
|---|---:|
| 生成圈子提问草稿 | 建议 P0 mock 或后端生成 |
| 保存草稿 | 可 P1 |
| 登录后发布到知乎圈子 | 可 P1 / P2，取决于知乎发布 API 可用性 |

未登录点击「去知乎圈子发布」时，前端需要展示：

```text
登录知乎后，才能发布到圈子
```

### 11.4 样本库小人卡

新版样本库不再使用圆形头像，而是使用小人卡。后端不用关心具体 SVG，但建议 `people[]` 保留一个可选形象字段：

| 字段 | 类型 | 用途 |
|---|---|---|
| `avatar` | string | 头像 URL 或 Open Peeps 素材 key |
| `characterImage` | string | 可选，小人卡形象 URL |
| `characterKey` | string | 可选，前端素材 key，例如 `openpeeps84.svg` |

如果后端不传，前端会用本地人物素材兜底。
